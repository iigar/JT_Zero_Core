# JT-Zero — About Project

## Що таке JT-Zero?

JT-Zero — це runtime система для автономних дронів на базі Raspberry Pi. Система забезпечує навігацію без GPS за допомогою Visual Odometry (візуальної одометрії) — камера дивиться вниз і вимірює зміщення дрона порівнюючи кадри.

**Архітектура:** C++ ядро (8 потоків реального часу) → pybind11 → FastAPI backend → React dashboard

**Підтримувані платформи:** Raspberry Pi Zero 2W (512MB), Pi 4 (4GB), Pi 5

**Польотний контролер:** Matek H743 з ArduPilot (MAVLink v2)

### Термальна камера — фронтально чи вниз?

| Монтаж | Застосування |
|--------|-------------|
| **Вниз** (рекомендовано) | Пошук людей/тварин зверху, інспекція дахів, сонячних панелей, термо-карта поля |
| **Фронтально** | Нічний політ (термальна бачить в темряві), пошук і порятунок (SAR), інспекція будівель збоку, виявлення пожеж |

**Рекомендація:** Термальну вниз для сканування, CSI вперед для навігації. Для VO навігації термальна камера не ідеальна (мало контрастних деталей), але працює як фолбек (тест: Det:180, Conf:0.18-0.29). CSI (visible light) завжди краща для VO. Якщо потрібно фронтальне термо — можна додати третій слот.

## Як працює Visual Odometry — повний цикл польоту

### Що таке Visual Odometry (VO)?

**Одометрія** — це вимірювання пройденого шляху. **Візуальна одометрія** — це коли камера "дивиться" на землю і розуміє, наскільки дрон зрушився, порівнюючи кадри.

Уявіть, що ви летите над полем і дивитесь вниз. Ви бачите камінь, кущ, стежку. Через секунду камінь зсунувся вліво на 10 пікселів у кадрі → значить дрон зсунувся вправо на X метрів.

### Повний цикл польоту

```
ЗЛІТ → НАБІР ВИСОТИ → МАРШРУТ → ПОВЕРНЕННЯ → ПОСАДКА
```

### Крок 1: Зліт

Дрон на землі. Система запускається:

```
1. C++ Runtime стартує 8 потоків
2. Camera Thread (T6) відкриває CSI камеру (rpicam-vid)
3. MAVLink Thread (T5) з'єднується з польотним контролером (Matek H743)
4. VO ініціалізується: перший кадр зберігається як "попередній"
5. Позиція = (0, 0, 0) — точка зльоту = точка відліку
```

### Крок 2: Як VO обчислює рух (кожні 66мс, 15fps)

```
Кадр N-1 (попередній)          Кадр N (поточний)
┌─────────────────────┐       ┌─────────────────────┐
│    ·  камінь         │       │         ·  камінь    │
│  ·  кущ              │  →→→  │       ·  кущ         │
│        · стежка      │       │             · стежка  │
└─────────────────────┘       └─────────────────────┘
                                 
  Камінь зсунувся на (dx, dy) пікселів →
  Дрон зрушився на (-dx, -dy) × масштаб
```

#### Алгоритм по кроках:

**a) Детекція точок (FAST-9 + Shi-Tomasi)**
- Шукаємо "цікаві" точки в кадрі — кути, контрастні плями
- FAST-9: швидкий детектор — перевіряє 16 пікселів по колу навколо кожної точки
- Shi-Tomasi: надійніший фолбек — обчислює "структурний тензор" (наскільки точка відрізняється від оточення)

**b) Трекінг точок (Lucas-Kanade Optical Flow)**
- Беремо точки з кадру N-1 і шукаємо їх в кадрі N
- LK алгоритм: "яка зміна (dx, dy) мінімізує різницю яскравості навколо точки?"
- Використовує градієнти (Sobel 3x3) і ітеративне уточнення до sub-pixel точності

**c) Фільтрація (Median + MAD)**
- Деякі точки трекнулись неправильно (шум, рух об'єктів)
- Рахуємо медіану зміщення всіх точок
- MAD (Median Absolute Deviation) — відкидаємо "викиди" (>2 sigma від медіани)
- Залишаються тільки "inliers" — надійні точки

**d) Обчислення реального зміщення**

```
dx_метри = dx_пікселів × висота / focal_length
dy_метри = dy_пікселів × висота / focal_length
```

- **focal_length** — оптична характеристика камери (задана для кожного сенсора)
- **висота** — від барометра або рейнджфайндера

**e) Фільтр Калмана**
- Згладжує швидкість, прибирає шум
- Порівнює з IMU (акселерометр) для перехресної валідації
- Якщо VO і IMU сильно розходяться → зменшуємо довіру

**f) Відправка в ArduPilot**

```
VISION_POSITION_ESTIMATE @ 25Hz → MAVLink → Matek H743 → EKF3
```

- ArduPilot EKF3 (Extended Kalman Filter) — "мозок" навігації
- EKF3 зливає разом: GPS + VO + IMU + Baro + Compass
- VO дає position estimate з коваріацією (чим менше довіра → більша коваріація)

### Крок 3: Маршрут (автономний політ)

```
Точка зльоту (0,0)──────→ Waypoint 1 (100м, 0м)
                                    │
                                    ↓
Waypoint 3 (0м, 200м) ←──── Waypoint 2 (100м, 200м)
```

- ArduPilot знає де дрон **зараз** (від EKF3 = GPS + VO + IMU)
- ArduPilot знає де **потрібно бути** (waypoint)
- Різниця = вектор руху → команди моторам
- **VO допомагає коли GPS поганий** (міська забудова, ліс, перешкоди)

### Крок 4: Повернення (RTL — Return To Launch)

```
Дрон на (523м, -89м, 50м) → Потрібно на (0, 0, 0)

EKF3 знає повну позицію завдяки:
- GPS: грубі координати (+-2м)  
- VO:  точне зміщення (+-0.1м на короткий час)
- IMU: мікрорухи між кадрами
- Baro: висота (+-0.5м)

Вектор додому = -поточна_позиція → летимо назад
```

**Проблема дрифту:** VO накопичує помилку з часом. За 10 хвилин польоту — можливо +-5м дрифту. Тому:
- GPS корегує абсолютну позицію
- VO дає точне "між кадрове" зміщення
- EKF3 оптимально зливає обидва

### Крок 5: Посадка

- Рейнджфайндер (ультразвуковий) дає точну висоту <2м
- VO стає дуже точним близько до землі (більше деталей, менший масштаб)
- Precision landing: VO бачить landingpad/маркер

## Навіщо VO якщо є GPS?

| Ситуація | GPS | VO |
|----------|-----|-----|
| Відкрите поле | Працює | Працює |
| Між будинками | Відбивання сигналу (+-10м) | Працює ідеально |
| Під мостом/навісом | Зникає | Працює |
| Ліс з густими деревами | Погано (+-5м) | Працює |
| Indoor (всередині будівлі) | Не працює | **Єдине джерело навігації** |
| Глушіння GPS (warfare) | Не працює | **Єдине джерело навігації** |

## Варіанти використання системи

### 1. Precision Agriculture (точне землеробство)
- Дрон з термальною камерою вниз
- VO для точного позиціонування над полем
- Термо-карта показує: вологість, хвороби рослин, зони поливу

### 2. Search & Rescue (пошук і порятунок)
- Нічний політ: термальна камера бачить людей
- VO навігація без GPS (ліс, гори, печери)
- Автономний пошук по сітці

### 3. Інспекція інфраструктури
- Мости, ЛЕП, вишки: VO тримає стабільну позицію для фото
- Термо: знаходить перегріті контакти, витоки тепла
- Сонячні панелі: знаходить дефектні комірки

### 4. Indoor навігація (склади, шахти)
- GPS = 0. VO — єдине джерело позиції
- Дрон може картографувати приміщення
- Автоматична інвентаризація складу

### 5. Картографія та 3D-моделювання
- Серія знімків з точними координатами від VO
- Photogrammetry → 3D модель території
- Ортофотоплан для будівництва

### 6. Охорона та патрулювання
- Автономний маршрут по периметру
- Термальна камера виявляє людей/тварин
- VO для стабільного польоту між будівлями
