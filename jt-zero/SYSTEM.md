# JT-Zero — Повний опис системи

## Навіщо створена ця система?

### Проблема

Коли дрон літає **всередині приміщення** (склад, завод, тунель, шахта) — GPS не працює. Без GPS польотний контролер (ArduPilot) не знає де знаходиться дрон і не може:
- Стабільно зависати на місці (дрейфує)
- Повернутися у точку зльоту (RTL)
- Летіти по маршруту (автономні місії)

Існують дорогі рішення: лідари ($500+), системи UWB ($1000+), камери Intel RealSense ($300+). Але вони важкі, дорогі, і не підходять для легких дронів.

### Рішення: JT-Zero

JT-Zero використовує **звичайну камеру Raspberry Pi** ($15) для визначення позиції дрона. Камера дивиться вниз, бачить текстуру поверхні (підлога, земля, трава), і обчислює переміщення дрона аналізуючи як "зсувається" картинка між кадрами.

Ці дані передаються на польотний контролер через **MAVLink** — стандартний протокол зв'язку дронів. ArduPilot приймає ці дані і використовує їх замість (або разом з) GPS.

**Результат:** дрон стабільно літає без GPS, використовуючи лише камеру за $15 і Raspberry Pi за $20.

### Для кого?

- **Дрон-ентузіасти** — indoor автономність без дорогого обладнання
- **Розробники** — платформа для експериментів з computer vision на дронах
- **Дослідники** — базова VO система для розширення (SLAM, mapping)
- **FPV пілоти** — додаткова стабілізація для indoor польотів

## Як працює система: алгоритм покроково

### Крок 1: Захоплення зображення

```
Pi Camera (OV5647) → CSI шлейф → Video4Linux2 → Grayscale кадр 320x240
```

Камера знімає 15 кадрів на секунду. Кожен кадр конвертується у чорно-білий (grayscale) формат — це прискорює обробку і зменшує використання пам'яті. Роздільна здатність 320x240 обрана спеціально для Pi Zero 2W: вища роздільна здатність не дасть кращої точності, але значно сповільнить обробку.

### Крок 2: Пошук характерних точок (Feature Detection)

```
Grayscale кадр → FAST Corner Detector → до 200 feature points
```

**Що таке feature point?** Це точка на зображенні де яскравість різко змінюється у кількох напрямках. Приклади: кут стола, край плитки, тріщина на асфальті, контрастний камінчик.

**Алгоритм FAST** (Features from Accelerated Segment Test):
1. Бере піксель і перевіряє 16 сусідніх пікселів по колу навколо нього
2. Якщо 12+ сусідів значно яскравіші або темніші — це feature (кут)
3. Швидкий: ~1 мс на кадр 320x240 на Pi Zero 2W

**Чому максимум 200 features?** Більше features = більше обчислень. На Pi Zero 2W обробка 200 features займає ~3 мс, а 500 features — вже ~10 мс. 200 — оптимальний баланс.

### Крок 3: Відстеження між кадрами (Feature Tracking)

```
Features з кадру [t-1] → Lucas-Kanade Optical Flow → Features в кадрі [t]
```

**Що це робить?** Для кожної feature з попереднього кадру шукає "ту саму" точку в новому кадрі. Результат: вектор переміщення (dx, dy) для кожної features в пікселях.

**Алгоритм Lucas-Kanade:**
1. Бере область 7x7 пікселів навколо feature
2. Шукає де ця область "найкраще підходить" у новому кадрі
3. Використовує ітеративний метод мінімізації різниці яскравості
4. Результат: точне переміщення в субпіксельній точності

**Tracking Quality:** Якщо з 200 знайдених features вдалося відстежити 140+ — tracking вважається якісним. Якщо менше 30 — tracking ненадійний і позиція "заморожується".

### Крок 4: Обчислення переміщення (Visual Odometry)

```
Набір векторів (dx, dy) → Медіанний фільтр → Δx, Δy в метрах
```

1. **Медіанний фільтр:** З усіх dx беремо медіану (середнє значення після сортування). Це відкидає "викиди" — features які відстежились неправильно.
2. **Конвертація пікселів у метри:**
   ```
   переміщення_метри = переміщення_пікселі × висота / фокусна_відстань
   ```
   - `фокусна_відстань` камери OV5647 ≈ 3.6 мм (в пікселях: ~1000)
   - `висота` береться з барометра або FC
3. **Накопичення позиції:**
   ```
   pose_x += Δx  (якщо tracking валідний)
   pose_y += Δy
   pose_z = -altitude_agl
   ```

### Крок 5: Передача на польотний контролер

```
Позиція + Орієнтація → MAVLink v2 → UART 115200 baud → Flight Controller
```

Три типи MAVLink повідомлень відправляються з Pi на FC:

| Повідомлення | ID | Частота | Що містить |
|---|---|---|---|
| **VISION_POSITION_ESTIMATE** | #102 | ~12 Hz | Позиція (x,y,z) + орієнтація (roll,pitch,yaw) |
| **ODOMETRY** | #331 | ~12 Hz | Позиція + швидкість + якість + кватерніон |
| **OPTICAL_FLOW_RAD** | #106 | ~6 Hz | Оптичний потік + відстань до землі |

### Крок 6: EKF Fusion

```
                    ┌──────────────────────┐
  VO позиція ──────→│                      │
  IMU дані ────────→│  Extended Kalman     │──→ Фінальна позиція
  Барометр ────────→│  Filter (EKF3)       │    (для стабілізації)
  GPS (якщо є) ───→│                      │
                    └──────────────────────┘
```

ArduPilot EKF3 об'єднує всі джерела даних і визначає найточнішу позицію. Якщо одне джерело тимчасово пропадає (наприклад, камера заблокована), EKF продовжує працювати на інших.

**Статус:** ArduPilot EKF3 успішно приймає VO дані від JT-Zero як **ExternalNav** джерело. Підтверджено повідомленням: `EKF3 IMU0 is using external nav`.

## Характеристики та обмеження

### Робочі параметри

| Параметр | Значення | Пояснення |
|----------|----------|-----------|
| **Максимальна швидкість** | 2-3 м/с | При вищій швидкості features "розмиваються" і tracking втрачається |
| **Рекомендована швидкість** | 0.5-1.5 м/с | Оптимально для стабільного tracking |
| **Висота польоту (min)** | 0.3 м | Нижче — камера бачить надто малу ділянку |
| **Висота польоту (max)** | 8-10 м | Вище — features стають надто дрібними |
| **Оптимальна висота** | 1-3 м | Найкращий баланс між точністю і покриттям |
| **Точність позиції** | ±5-20 см | Залежить від текстури поверхні та висоти |
| **Дрейф** | 1-5% від шляху | Накопичується з часом (VO без loop closure) |
| **Затримка (latency)** | 50-80 мс | Від захоплення кадру до відправки на FC |
| **Частота оновлення** | ~12 Hz | Достатньо для ArduPilot EKF |
| **Камера FPS** | 15 кадрів/с | Оптимально для Pi Zero 2W |
| **Max features** | 200 | Обмеження для швидкості обробки |
| **Дальність польоту** | Необмежена* | *З накопиченням дрейфу ~1-5% від пройденого шляху |

### Приклад дрейфу

Якщо дрон пролетів 100 метрів по прямій і повернувся назад:
- **Ідеально:** повинен опинитися в точці старту (0, 0)
- **Реально:** буде зміщення 1-5 метрів (це і є дрейф 1-5%)
- **З GPS fusion (outdoor):** дрейф зменшується до сантиметрів

### Що впливає на якість?

| Фактор | Добре | Погано |
|--------|-------|--------|
| **Поверхня** | Текстурована (плитка, килим, асфальт) | Однорідна (бетон, сніг, вода) |
| **Освітлення** | Рівне, яскраве | Мерехтливе, темне |
| **Рух** | Плавний, повільний | Різкий, швидкий |
| **Висота** | 1-3 м | >8 м або <0.3 м |
| **Вібрації** | Мінімальні (демпфер) | Сильні (без демпфера) |

### Обмеження системи

| Обмеження | Причина | Можливе рішення |
|-----------|---------|-----------------|
| **Дрейф позиції** | VO накопичує помилки без корекції | GPS fusion outdoor, loop closure indoor |
| **Погане освітлення** | Камера не бачить features | LED підсвітка або IR камера |
| **Однорідні поверхні** | Немає features для tracking | Текстурований килимок, наклейки на підлозі |
| **Різкі повороти** | Features "зникають" з поля зору | Обмежити швидкість повороту в ArduPilot |
| **Вібрації** | Розмиття кадрів (motion blur) | Гумовий демпфер для камери |
| **Тіні та відблиски** | Хибні features | Рівне освітлення, поляризаційний фільтр |

## Архітектура системи

### Загальна схема

```
┌─────────────────────────────────────────────────────────────────────┐
│                         Raspberry Pi Zero 2 W                       │
│                                                                     │
│  ┌──────────┐    ┌────────────────────────────────────────────────┐ │
│  │ Pi Camera│───→│              C++ Ядро (jtzero_core)            │ │
│  │ (CSI)    │    │                                                │ │
│  └──────────┘    │  T1: Sensors (200Hz) — IMU, Baro, GPS, Range  │ │
│                  │  T2: Events  (200Hz) — диспетчер подій         │ │
│                  │  T3: Reflex  (200Hz) — швидкі реакції <5ms     │ │
│                  │  T4: Rules   (20Hz)  — поведінкова логіка      │ │
│                  │  T5: MAVLink (50Hz)  — зв'язок з FC            │ │
│                  │  T6: Camera  (15fps) — VO обробка              │ │
│                  │  T0: Supervisor (10Hz) — здоров'я системи      │ │
│                  │                                                │ │
│                  └───────────────┬────────────────────────────────┘ │
│                                 │ pybind11                         │
│                  ┌──────────────┴─────────────────────────────┐    │
│                  │         Python Backend (FastAPI)            │    │
│                  │  REST API + WebSocket (10Hz) + Camera PNG   │    │
│                  └──────────────┬─────────────────────────────┘    │
│                                 │                                  │
│                  ┌──────────────┴─────────────────────────────┐    │
│                  │         React Dashboard (браузер)           │    │
│                  │  7 вкладок: Dashboard, Telemetry, Camera,   │    │
│                  │  MAVLink, Events, Docs, Settings            │    │
│                  └────────────────────────────────────────────┘    │
│                                                                     │
│  ┌──────────┐                                                       │
│  │ UART TX  │───→  MAVLink v2 → Flight Controller (ArduPilot)      │
│  │ UART RX  │←───  Telemetry  ← (Matek H743, Pixhawk, etc.)       │
│  │ GND      │────  Ground     ─                                     │
│  └──────────┘                                                       │
└─────────────────────────────────────────────────────────────────────┘
```

### C++ ядро: 8 потоків реального часу

C++ ядро працює у 8 паралельних потоках. Кожен потік має свою частоту і пріоритет:

| Потік | Назва | Частота | Що робить |
|-------|-------|---------|-----------|
| **T0** | Supervisor | 10 Hz | Моніторинг здоров'я: температура CPU, напруга батареї, стан потоків. Запис телеметрії в кільцевий буфер. |
| **T1** | Sensors | 200 Hz | Читання даних з усіх сенсорів (IMU, барометр, GPS, далекомір, оптичний потік). Автовизначення: hardware → MAVLink → simulation. |
| **T2** | Events | 200 Hz | Диспетчер подій. Lock-free черга на 1024 події. Розподіляє події між reflex та rule engine. |
| **T3** | Reflex | 200 Hz | Швидкі реакції з гарантією <5 мс: аварійна зупинка, попередження про перешкоди, захист від перевороту. |
| **T4** | Rules | 20 Hz | Складна поведінкова логіка: автоматичне повернення при низькій батареї, утримання позиції при втраті GPS. |
| **T5** | MAVLink | 50 Hz | Прийом телеметрії від FC (attitude, GPS, battery). Відправка VO даних на FC (VISION_POSITION_ESTIMATE, ODOMETRY). |
| **T6** | Camera | 15 FPS | Захоплення кадрів, FAST detection, LK tracking, обчислення VO. |
| **T7** | API | 30 Hz | HTTP/WebSocket бридж для React Dashboard. |

### Потік даних

```
Камера → Feature Detection → Feature Tracking → Visual Odometry
   │                                                    │
   │                                                    ▼
   │                                         MAVLink серіалізація
   │                                                    │
   │                                                    ▼
   │                                          UART → FC (ArduPilot)
   │                                                    │
   ▼                                                    ▼
WebSocket → React Dashboard              EKF3 Fusion → Стабільний політ
```

### Режими роботи сенсорів

JT-Zero автоматично визначає яке обладнання доступне:

| Режим | Колір в UI | Коли використовується |
|-------|-----------|----------------------|
| **HARDWARE** (HW) | Зелений | Прямий I2C/SPI драйвер (MPU6050, BMP280) |
| **MAVLINK** (MAV) | Блакитний | Дані від FC через MAVLink (основний режим на польотному контролері) |
| **SIMULATED** (SIM) | Жовтий | Програмна симуляція (для розробки без обладнання) |

Пріоритет: **hardware → mavlink → simulated**

## Перевірена конфігурація

Система протестована і працює з наступним обладнанням:

- **Pi:** Raspberry Pi Zero 2 W (Raspbian Bookworm 64-bit)
- **Камера:** Pi Camera v2 (OV5647), CSI, 15 FPS @ 320x240
- **FC:** Matek H743-SLIM V3, ArduCopter V4.3.6
- **З'єднання:** UART (GPIO14 TX → SERIAL4 RX, GPIO15 RX → SERIAL4 TX)
- **Baud rate:** 115200
- **EKF:** EKF3, ExternalNav як джерело позиції
- **VO Rate:** ~12 Hz (VISION_POSITION_ESTIMATE + ODOMETRY)
- **Результат:** `EKF3 IMU0 is using external nav` — EKF приймає VO дані
